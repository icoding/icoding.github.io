前言

Optional现在大家应该都很熟悉，java8现在就已经支持Optional了，但是在jdk6、jdk7的年代，jdk是没有这样的处理方式的，google就超前的提供了Optional的支持。
其实不仅仅是Optional，包括Predicate、函数式编程等，都是在java8中一一得到内置的功能。不得不说，google这家公司真是厉害。

Optional解决的问题

用一句话来说明Optional真正的要解决的问题那就是：

让我们来理解下这句话的意思：

大家都听过Optional是用来解决空指针异常的，那么为什么会出现空指针异常呢？
因为除了基本类型，其他的类型对象在初始化的时候就是null，如果忘记对该对象赋值，或者因为其他原因没能正确赋值，那么在使用对象操作其方法的时候就会出现最常见的空指针异常。

为什么Optional就解决了空指针异常呢？
其实Optional也没有解决这个异常，只是把对象封装了，你可以通过api的方式知道他是null或者不是null，而不是用以前的 if(obj == null) 这种样板代码来污染你整洁的业务逻辑。
所以我觉得与其说他解决了空指针异常的问题，倒不如说是用来引导程序员主动思考。
因为如果你看到了某个方法返回的对象是Optional封装的，那么他肯定是会有null的情况，你需要去做相应的处理，而不是像以前那样看见一个对象根本就没有想过他是否为null而直接去使用，或者机械的不管会不会返回null都去增加if的判断。

说到这里，大家应该对Optional有些了解了，起码知道如果发现某个程序返回了Optional或者传入了Optional，一定会有“嗯、、这个对象有猫腻，哥要好好弄弄他、、”这个心理活动。

orElse 和 orElseGet 的区别
这两个方法看似相同，都是为了在内部对象为null的时候获取一个默认值，
但是当内部对象不为null到时候他们的内部是孑然不同的机制.
orElse 每次都会调用传入的方法，而orElseGet则不会每次调用。二者在此会有性能上的差异。
可以看到 orElseGet和guava的or(supplier)是一样的

使用场景，java8、guava区别？

Optional解决的问题大家已经知道了，所以最佳使用地点其实也是很清晰，那就是方法的返回值。

既然java8现在已经支持了Optional，为什么还要使用guava呢？
他们有个最大的区别就是guava中的Optional是可系列化的，而java中的是不可序列化的。
所以java的Optional是不能作用在字段上的，而guava的则可以在字段上作用。
